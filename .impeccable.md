# Threadline design context

This document records the confirmed design brief that guides every interface
decision in Threadline.

## Design context

### Users

Threadline serves synthetic patient and clinician personas in a hackathon demo.
Patients arrive reflective or anxious and need continuity without restating
their history. Clinicians need fast, auditable review instead of another inbox.

### Brand personality

The product is calm, clinical, and careful. It communicates warmth without
becoming playful, and technical confidence without implying medical authority.

### Aesthetic direction

Use a light, editorial care-workspace aesthetic. Warm tinted paper surfaces,
muted mulberry accents, readable typography, and an asymmetric memory thread
make the interface recognizable. Avoid chatbot chrome, generic card grids,
gradient text, decorative glass, neon AI colors, and colored side stripes.

### Design principles

- Put continuity before novelty: the remembered detail is the focal point.
- Make every memory inspectable, reviewable, and reversible.
- Use quiet surfaces and precise hierarchy to lower cognitive load.
- Show technical evidence without exposing private prompt content.
- Adapt the workflow on mobile without hiding critical controls.
- Use motion only for state explanation and direct feedback.

## Feature brief

Threadline lets a patient reflect with Qwen, converts the session into proposed
memories, routes those memories through clinician review, and visibly reuses
approved context in a later session. The primary action is completing the
two-session continuity loop while understanding exactly what was remembered.

The patient session uses a conversation column paired with a Memory Trace. The
clinician workspace emphasizes a review queue and patient timeline. Empty,
loading, streaming, finalizing, failed, approved, disputed, forgotten, and
high-risk states all provide explicit next actions and persistent status text.
